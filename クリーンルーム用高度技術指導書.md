# クリーンルーム用高度技術指導書

## 1. 概要
本書は、契約書管理システムの設計および実装において使用されている高度なセキュリティ技術およびアーキテクチャについて、技術者向けに詳細な仕様を解説するものです。本システムは隔離された安全な環境（クリーンルーム）で扱うべき機密情報を処理するため、データ保護と証拠保全において厳格なアプローチを採用しています。

## 2. 暗号化アーキテクチャ

### 2.1 パスワードのハッシュ化 (PBKDF2)
- **アルゴリズム**: HMAC-SHA256 をベースとした PBKDF2 (Password-Based Key Derivation Function 2)
- **ソルト**: アカウント作成時にユーザーごとに一意のランダムな16バイトのソルトを暗号論的疑似乱数生成器により生成。
- **イテレーション回数**: 300,000回（総当たり攻撃およびレインボーテーブル攻撃に対する十分な耐性を確保）。
- **用途**: データベース漏洩時におけるパスワードの復元を事実上不可能にするため。

### 2.2 本人確認書類のハイブリッド暗号化 (ECC + AES-256-GCM)
本人確認書類（画像データなど）は、管理者の公開鍵を用いて非対称暗号ベースのハイブリッド暗号方式により保護されます。

- **非対称暗号 (ECDH)**: 
  - 曲線: NIST P-256 (secp256r1)
  - 管理者側（Admin）で永続的なECC鍵ペアを生成し、公開鍵のみをシステム内で利用可能にする。
  - クライアント側（User）で画像アップロード時に一時的（Ephemeral）なECC鍵ペアを生成し、管理者の公開鍵と組み合わせて ECDH (Elliptic Curve Diffie-Hellman) 鍵共有により共有シークレット（Derived Key）を導出。
- **対称暗号 (AES-GCM)**: 
  - アルゴリズム: AES-256-GCM (Galois/Counter Mode)
  - 導出された共有シークレットを鍵として用い、画像データを暗号化。
  - 認証付き暗号であるため、16バイトの認証タグ (Tag) によりデータの完全性（改ざんされていないこと）を保証。
  - 暗号化されたペイロード構成: `Nonce (12 Bytes) || Tag (16 Bytes) || Ciphertext`
- **保存プロセス**: 暗号化された画像データと、クライアント側の一時的な公開鍵（Base64形式）を合わせてJSON形式のデータストアに保存。管理者の秘密鍵を持たない限り、いかなるユーザー（アップロードした本人含む）も画像を復元できない。

### 2.3 音声データと電子署名の完全性確保 (SHA-256)
事前説明確認時の録音データおよび署名データについて、データの改ざんを検知するために SHA-256 (Secure Hash Algorithm 256-bit) によりハッシュ値を計算して記録。これにより、後日「説明時の内容が改変された」などの不正を防止します。

## 3. 実装アーキテクチャとデータフロー

### 3.1 アプリケーション構造 (WPF)
- **UI層**: Windows Presentation Foundation (WPF) を使用し、MVVMパターンに基づくXAMLで構築。画面サイズの動的変更に対応。
- **サービス層**: `AuthService`、`ContractService`、`UserService`、`CryptoService`、`AudioRecordService` など、ビジネスロジックを分離した静的サービスクラスとして実装。
- **データ層**: `DataStore` クラスを介してローカルの JSON ファイル群と連携。データベースエンジンを不要とし、ポータビリティと監査のしやすさを向上。

### 3.2 状態遷移 (ステートマシン)
契約ステータスは列挙型 `ContractStatus` により厳密に管理され、以下のような不可逆・可逆の制約を持ちます。
- `Pending` から `Agreed` または `Rejected` への遷移。
- `Agreed` になった契約は原則変更不可。
- 「代理同意」の場合に限り、不完全な合意であるとして消費者側から `Terminated` への遷移を許可するフォールバックフローを備える。
- `ResolveDispute` 操作によって、議論完了後は強制的に `Pending` にリセットさせ、再署名を要求する安全設計。

## 4. 運用上の注意点
- 管理者のECC秘密鍵 (`admin_ecc_key.json`) は本システムの信頼性の根幹であり、クリーンルーム外への持ち出しは厳禁とします。
- `System.Text.Json` などのサードパーティ依存関係は、既知の脆弱性（例: GHSA-8g4q-xg66-9fp4）を排除するため常にパッチ適用の最新版を使用してください。
